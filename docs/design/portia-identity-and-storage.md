# Portia Identity and Storage

## 1. Current Core Identity and Storage Constraints

Portia’s identity and storage model must begin with the contracts already implemented by `pds-core`.

This section records the current Core behavior without deciding Portia’s final model. Later sections will evaluate whether Portia can operate entirely within those contracts or whether Portia requires a new module-specific convention or a shared Core extension.

## 1.1 Core’s Ownership Boundary

Core owns shared suite infrastructure rather than module-specific educational records.

Its current responsibilities include:

* workspace-root resolution and validation;
* workspace baseline setup;
* shared class-folder paths;
* shared roster parsing and validation;
* class-level metadata;
* student display helpers;
* shared identifier validation;
* active school-year state;
* module-qualified work identity;
* PDS2 route identity;
* route-registration storage;
* and safe construction of module-owned descendant paths.

Modules remain responsible for:

* the meaning of their `work_id`;
* their domain records;
* their domain-specific identifiers;
* the organization of records beneath a module work root;
* indexes and reports;
* correction and lifecycle semantics;
* and teacher-facing workflows.

Core does not define a universal:

* assignment record;
* student submission;
* activity;
* Event;
* support case;
* intervention;
* result;
* report;
* or student dossier.

Portia must consume Core’s shared identity and path contracts without treating Core as the owner of Portia’s behavior-support domain.

## 1.2 Workspace Root and Workspace Identity

Core resolves one selected filesystem directory as the Paper Data Suite workspace root.

Resolution follows this precedence:

```text
1. explicit runtime argument
2. PDS_WORKSPACE_ROOT environment variable
3. saved user configuration
4. ~/Paper Data Suite
```

The saved preference exists outside the workspace. Changing or clearing that preference does not:

* move files;
* copy files;
* merge workspaces;
* delete the old workspace;
* or establish that two paths identify the same workspace.

When Core initializes a workspace, it creates a small marker:

```text
<PDS workspace>/.pds/workspace.json
```

The current marker contains only:

```json
{
  "created_by": "pds-core",
  "workspace_version": 1
}
```

It does not contain a durable:

* `workspace_id`;
* organization identifier;
* school identifier;
* teacher identifier;
* tenant identifier;
* or creation identity suitable for cross-device reconciliation.

The operative identity of the workspace is therefore its selected filesystem root. Core can normalize and validate that path, but it does not provide a stable logical identity that survives:

* moving the workspace;
* copying it;
* restoring it from backup;
* synchronizing it to another device;
* or combining records from several workspaces.

Core creates these shared baseline locations:

```text
<PDS workspace>/
  .pds/
    workspace.json
  classes/
  scans_inbox/
  scans/
    source/
    review/
```

It does not create a generic workspace-level module collection such as:

```text
<PDS workspace>/modules/<module_id>/
```

The audited Core contracts therefore do not currently provide a canonical workspace-scoped Portia root.

A Portia design using workspace-scoped canonical records would require either:

1. a Portia-owned path convention not currently supplied by Core; or
2. a new shared Core contract for workspace- or organization-scoped module storage.

The later sections resolve that choice for Portia v1.

## 1.3 Shared Identifier Rules

Core’s generic identifier validator permits:

```text
ASCII letters
digits
underscore
hyphen
```

Identifiers must:

* be strings;
* be nonempty;
* contain no leading or trailing whitespace;
* and contain no other characters.

Core rejects invalid values rather than silently normalizing them.

Additional contracts may impose narrower rules. For example:

* `module_id` must be lowercase;
* `record_kind` in a `ModuleRecordRef` must be lowercase;
* route IDs generated by Core use a nonsemantic random format.

The generic validator establishes path and QR safety. It does not establish semantic safety.

A value can satisfy Core’s identifier rules while still exposing inappropriate information. For example, the generic validator would not itself prohibit an identifier containing:

* a student number;
* an abbreviated student name;
* a behavior category;
* an allegation;
* or a support status.

Portia must therefore impose a stronger rule:

> Portia identifiers must be opaque and must not encode student identity, behavior meaning, disability information, allegations, determinations, or other sensitive semantics.

Core’s safe-character policy is necessary but not sufficient for Portia record identity.

## 1.4 Class Identity

Core represents a class through a validated `class_id` and a canonical class folder:

```text
classes/<class_id>/
```

The canonical class-level files are:

```text
classes/<class_id>/
  roster.csv
  class.json
```

Within one selected workspace, `class_id` is the path key used to locate one class folder.

Core does not currently compose class identity from:

```text
school_year + class_id
```

The school year is stored inside `class.json`, not in the class path.

Consequently, these two conceptual classes cannot coexist at distinct canonical Core paths merely because their metadata has different school years:

```text
2025-2026 / english10_p2
2026-2027 / english10_p2
```

Both would resolve to:

```text
classes/english10_p2/
```

A teacher who needs both classes present in the same active workspace must therefore use distinct `class_id` values, archive or move the earlier class through a future lifecycle process, or rely on a future change to the shared class-identity contract.

### Class Metadata

Core’s strict class metadata record contains:

```json
{
  "schema_version": "1",
  "record_type": "class",
  "class_id": "english10_p2",
  "school_year": "2026-2027",
  "created_at": "2026-07-09T13:00:00-04:00",
  "updated_at": "2026-07-09T13:00:00-04:00",
  "module_details": {}
}
```

The metadata:

* describes the class as a whole;
* is separate from the roster;
* must match the containing class folder when loaded through canonical helpers;
* is written explicitly;
* and is not created automatically when a roster is loaded or written.

Existing class folders without `class.json` remain valid. A caller must explicitly request metadata-aware discovery or loading.

Therefore:

* a class may have no recorded school year;
* an active workspace school year does not automatically become the class school year;
* and Portia cannot assume that every existing Core class has complete temporal metadata.

## 1.5 Roster Identity

Core’s canonical roster is:

```text
classes/<class_id>/roster.csv
```

Its required columns are:

```text
class_id
student_id
last_name
first_name
period
```

A valid roster must:

* contain at least one student;
* contain one exact `class_id` across every row;
* contain no duplicate `student_id` within that roster;
* preserve identifier values as strings;
* preserve leading zeros;
* and validate `class_id` and `student_id` through Core’s identifier rules.

A Core `Roster` is an immutable collection associated with one class.

A Core `StudentRecord` contains:

```text
class_id
student_id
last_name
first_name
period
extra_fields
```

Optional roster columns are retained as `extra_fields`, but Core does not interpret them automatically or require modules to export them.

### Scope of `student_id`

The current Core contract states:

> `student_id` is the canonical identity within a `Roster` and for roster lookups.

This scope is narrower than workspace-wide or organization-wide identity.

Core rejects duplicate student IDs within one roster. The audited public contracts do not:

* reject the same `student_id` appearing in several rosters;
* require the same `student_id` in several rosters to identify the same person;
* verify that names or other metadata agree across rosters;
* create a workspace-level student record;
* expose a cross-roster identity registry;
* or resolve several roster entries into one authoritative person identity.

Therefore, matching `student_id` values across two rosters are not, under the current formal contract, sufficient proof that the records identify the same person.

They may represent the same student by local convention, but Core does not currently establish or validate that relationship.

Likewise, different `student_id` values do not prove that two roster records identify different people. They may reflect:

* separate imports;
* changed local identifiers;
* data-entry differences;
* transfers;
* or inconsistent source systems.

Portia must not merge or separate people automatically from these values without an accepted cross-roster identity decision.

### Display Names

Core treats names as display information.

`first_name`, `last_name`, and optional `preferred_name` are not durable identity.

A preferred name may alter teacher-facing display but must not replace `student_id` or become a hidden identifier.

Portia must not use:

* full name;
* sort name;
* preferred name;
* initials;
* or a normalized name string

as a durable relationship key.

## 1.6 Class Discovery and Missing Data

Core discovers classes by examining directories beneath:

```text
<PDS workspace>/classes/
```

Invalid directory names are skipped rather than reinterpreted.

Class discovery may optionally require or load:

* `roster.csv`;
* `class.json`;
* both;
* or neither.

When loading is requested, invalid roster or metadata records cause that class to be skipped by the listing helper.

This creates an important distinction:

```text
class folder exists
≠ valid roster exists
≠ valid class metadata exists
```

Portia must decide its own minimum requirements for writable workflows.

For example, Portia may require:

* a valid class folder;
* a valid roster;
* valid class metadata;
* an active school year;
* or some defined combination.

That requirement must be explicit. Portia should not treat a discovered class directory as proof that student or school-year identity is complete.

## 1.7 Module-Qualified Work Identity

Core’s current generic work identity is:

```text
ModuleWorkRef
  = module_id
  + class_id
  + work_id
```

Its canonical storage root is:

```text
classes/<class_id>/modules/<module_id>/work/<work_id>/
```

A bare `work_id` is not suite-wide identity.

The same `work_id` may be reused:

* in another class;
* by another module;
* or by another module in the same class.

For example, these are distinct:

```text
scoreform / english10_p3 / project_check
quillan   / english10_p3 / project_check
concord   / english10_p3 / project_check
```

Core deliberately does not define `work_id` as an assignment ID.

Its meaning belongs to the module. Existing examples include:

* ScoreForm assignment;
* Quillan assignment;
* Concord Activity;
* another module’s durable top-level work context.

Core does not require one work context to:

* belong to one student;
* represent graded work;
* be an assessment;
* be a submission;
* or be reportable to a gradebook.

Portia must therefore make an explicit decision about what its `work_id` means.

At the audit stage, possible Portia meanings included:

* Event;
* support process;
* intervention;
* student-year context;
* case;
* or another top-level aggregate.

Core supplies no default answer. Section 3 records Portia’s selected meaning.

### Class Qualification Is Mandatory in the Current Core Work Contract

Every current `ModuleWorkRef` requires one `class_id`.

Core does not currently expose a generic equivalent such as:

```text
WorkspaceModuleWorkRef
OrganizationModuleWorkRef
CrossClassWorkRef
```

Nor does it expose a canonical path such as:

```text
modules/portia/work/<work_id>/
```

outside a class.

Therefore, a Portia record that has no legitimate owning class cannot be placed into a current Core module work root without either:

* selecting a class;
* defining a new Portia-specific root;
* or extending Core.

Selecting an arbitrary class would misrepresent ownership and is prohibited by Portia’s existing decisions.

## 1.8 Module-Owned Descendants

Core owns the canonical module work root. The module owns the organization and meaning of descendants beneath it.

For example:

```text
classes/<class_id>/modules/portia/work/<work_id>/
  event.json
  accounts/
  participants/
  supports/
  follow_ups/
  outcomes/
  indexes/
  exports/
```

would be structurally possible once Portia defines the semantics.

Core’s `safe_module_work_descendant(...)` helper prevents lexical path escape through:

* absolute paths;
* `..`;
* `.` components;
* empty path components;
* drive-qualified paths;
* UNC paths;
* or mixed slash traversal.

The helper does not:

* create the path;
* validate a Portia record;
* establish record ownership;
* follow or reject existing symlinks;
* or guarantee runtime containment against an untrusted symlink.

Portia must define its own:

* directory-creation behavior;
* atomic-write rules;
* collision behavior;
* canonical versus derived files;
* repair behavior;
* and stronger containment rules where untrusted filesystem contents are possible.

## 1.9 Route Identity and Domain Record Identity

Core separates several forms of identity:

```text
ModuleWorkRef
RouteLocator
RouteRegistration
ModuleRecordRef
student relationship
Author
Subject
final evidence destination
```

They are related only through explicit records. Core does not infer one from another.

### PDS2 Route Identity

A PDS2 locator is:

```text
module_id + class_id + work_id + route_id
```

Its serialized form is:

```text
PDS2|m=<module_id>|c=<class_id>|w=<work_id>|r=<route_id>
```

PDS2 applies only to a physical page expected to return through scanning.

A digitally entered Portia Event, Account, Support, Follow-Up, or Outcome does not need a PDS2 identity.

### Route Identity Remains Class-Scoped

Every PDS2 locator contains one `class_id`.

Therefore, if Portia later produces returned paper, each returned page route must have a legitimate class-qualified work context under the current Core contract.

This remains true even when the underlying Portia process:

* involves several students;
* concerns several classes;
* or refers to a workspace-scoped canonical record.

### Route Targets Are Separate

A Core `ModuleRecordRef` contains:

```text
module_id
record_kind
record_id
contract_version
```

It does not contain:

* `class_id`;
* `work_id`;
* a filesystem path;
* student identity;
* Author identity;
* or Subject identity.

Core validates the reference structure but does not:

* locate the target;
* require its file to exist;
* interpret its record type;
* or require it to be stored beneath the route’s work root.

A route registration requires the locator and target to belong to the same module, but the owning module determines the target’s actual semantic record and storage.

This separation is significant for Portia.

A future Portia page route could be class-qualified while resolving to a Portia-owned record whose complete participant and ownership relationships are defined elsewhere. Core routing does not itself settle where that canonical Portia record must live.

However, any such model would require an explicit Portia storage contract. Core will not infer it.

## 1.10 Active School-Year State

Core maintains active school-year state at:

```text
<PDS workspace>/settings/school_year.json
```

The state contains:

```text
active_school_year
opened_at
closed_at
```

School-year values must use consecutive years:

```text
YYYY-YYYY
```

For example:

```text
2026-2027
```

The state may be:

* missing;
* open;
* or closed.

`get_active_school_year(...)` returns:

* the school year when the state is open;
* `None` when the state file is absent;
* and `None` when the stored state is closed.

Opening or closing a school year does not:

* create classes;
* create rosters;
* create module work;
* archive records;
* move files;
* delete files;
* create reports;
* or migrate data.

The active school-year record is therefore current workspace context, not a storage container or archival boundary.

### Active State and Class Metadata Are Independent

Class metadata stores its own `school_year`.

Core does not automatically:

* copy the active school year into `class.json`;
* change class metadata when the active year changes;
* move a class when a year closes;
* or verify that every class belongs to the currently active year.

This independence is intentional because low-level roster and class operations avoid hidden side effects.

Portia must therefore use school-year values explicitly.

It cannot assume:

```text
current active school year
= school year of every selected class
= school year of every historical Portia record
```

Each durable Portia record will need its own temporal relationship or an unambiguous relationship to a class or parent record that supplies that context.

## 1.12 Capabilities Core Does Not Currently Provide

The audited Core contracts do not currently provide:

* a durable `workspace_id`;
* school or organization identity;
* a workspace-level canonical student registry;
* cross-roster student resolution;
* a reviewed identity-link record between roster entries;
* institution-wide person identity;
* staff, family, or visitor identity;
* a generic workspace-scoped module root;
* an organization-scoped module work reference;
* a cross-class module work reference;
* multi-student Event ownership;
* a generic domain-record ID generator for arbitrary module records;
* Portia-specific canonical record paths;
* Portia-specific indexes or projections;
* cross-year Portia continuation;
* school-year archival or rollover;
* or authenticated user identity.

These are not defects in the existing Core contract. They are outside the scope Core has implemented so far.

Issue #4 must determine which missing capabilities:

1. are unnecessary for the first Portia release;
2. can be defined safely as Portia-owned behavior;
3. require a shared Core extension;
4. or must remain deferred.

## 1.13 Constraints Established by the Audit

The following constraints should govern the remainder of this design.

1. **A Core `student_id` must initially be treated as roster-scoped.**

   Portia must not assume workspace-wide identity merely because two rosters contain matching values.

2. **Names cannot resolve cross-roster identity.**

   Matching names, preferred names, initials, or sort names are insufficient.

3. **`class_id` currently identifies a class folder within one workspace.**

   School year is metadata, not part of the class path identity.

4. **Active school-year state is not record identity.**

   It cannot substitute for an Event occurrence year, support effective period, or class metadata.

5. **Current Core module work is class-qualified.**

   A record with no legitimate class owner cannot be forced into a Core work root without an additional decision.

6. **Portia must define the meaning of `work_id`.**

   Core intentionally leaves that meaning to the module.

7. **A canonical Portia record must have one authoritative location.**

   Class, student, or work indexes must not become independently editable copies.

8. **Cross-class cases cannot be assigned to an arbitrary class.**

   Doing so would conflate storage convenience with domain ownership.

9. **A synthetic `schoolwide` class is not compatible with honest Core identity.**

   It would create a class-shaped record for something that is not a class.

10. **A workspace-scoped or hybrid Portia design is not currently a standard Core path.**

    It would require a documented Portia convention or a Core enhancement.

11. **PDS2 does not determine Portia’s digital storage model.**

    It applies only to expected returned pages and separates route identity from semantic record identity.

12. **Any Portia identifier policy must be stricter than Core’s generic safe-character validation.**

    Portia IDs and paths must not expose sensitive semantics.

13. **Class and roster completeness must be tested explicitly.**

    A class folder may exist without a valid roster or class metadata.

14. **Cross-year continuity must be modeled through explicit record relationships.**

    Core does not automatically roll classes, students, or module records forward.

15. **Any new shared identity authority must be owned deliberately.**

    Portia must not create a competing suite-wide roster or person registry as an incidental implementation detail.

* whether matching `student_id` values may be linked across rosters;

## 2. Portia’s Required Identity Layers

Portia’s identity model should serve the actual Paper Data Suite product boundary:

> Paper Data Suite is a local-first suite adopted and operated by an individual teacher. Portia records belong to that teacher’s classroom practice rather than to a schoolwide or district-wide behavior-management system.

Portia v1 therefore does not require:

* organization identity;
* school identity;
* district identity;
* institution-wide student identity;
* cross-teacher identity reconciliation;
* schoolwide Event ownership;
* ownership of one Event or Support Process by several classes;
* or a multi-user institutional person directory.

Every Event, Support Process, and subordinate workflow record belongs to exactly one owning Core class. That ownership does not require every student participant or referenced Event to come from the owning class’s roster. Portia may use explicit roster-qualified references to other classes taught by the same teacher.

Workspace-scoped Actor records are the sole Portia v1 exception to class-owned canonical workflow storage. They belong to the selected teacher workspace because the same recurring non-roster person may participate across several classes.

A broader institutional model would require a whole-suite architectural redesign rather than an extension of Portia’s initial identity model.

---

## 2.1 Identity Model Overview

Portia requires the following distinct identity and context layers:

| Layer                         | Initial authority                                | Purpose                                                                                  |
| ----------------------------- | ------------------------------------------------ | ---------------------------------------------------------------------------------------- |
| Workspace context             | Core workspace root                              | Determines the selected teacher-local Paper Data Suite workspace                         |
| School-year context           | Core class metadata                              | Establishes the academic year associated with class-owned work                           |
| Class identity                | Core `class_id`                                  | Identifies the one class that owns an Event or Support Process                            |
| Roster identity               | Canonical Core roster identified by `class_id`   | Supplies the authoritative source for each roster-qualified student reference            |
| Student reference             | `class_id + student_id`                          | Identifies one student within the reference’s source roster                               |
| Actor identity                | Opaque Portia `actor_id`                         | Identifies one recurring non-roster person in the teacher-local Actor Directory           |
| Local operator attribution    | Portia configuration or record metadata          | Identifies the teacher-entered source without claiming authentication                    |
| Statement-source attribution  | Student reference, Actor reference, or metadata  | Records who supplied an Account or statement                                             |
| Portia work identity          | `module_id + class_id + work_id`                 | Identifies one top-level class-owned Portia workflow object                               |
| Portia domain-record identity | Opaque Portia record ID                          | Identifies Events, Accounts, Supports, Follow-Ups, Outcomes, and other records           |
| Event-participant identity    | Opaque participant-relationship ID               | Identifies one subject’s relationship to one Event                                        |
| External-record identity      | Typed module-qualified reference                 | Identifies a record owned by Core or another module                                      |
| Returned-page route identity  | Core PDS2 locator                                | Identifies one expected returned Portia paper page                                       |

These layers must remain separate. No single identifier should be made to perform all of these roles.

---

## 2.2 Workspace Context

Portia operates inside the Core-selected workspace root.

The workspace root determines:

* where classes are discovered;
* where Portia work is stored;
* which active school-year state is available;
* and which local Core configuration is being used.

The workspace path is operational context, not a durable Portia record identifier.

Portia records must not store absolute workspace paths as identity. Doing so would make records fragile when a teacher:

* moves the workspace;
* restores it from backup;
* changes computers;
* or places it inside a synchronized folder.

All persisted Portia relationships should use durable identifiers and workspace-relative contracts rather than absolute filesystem paths.

Because Core currently has no durable `workspace_id`, Portia v1 does not attempt to reconcile or merge records from different workspace copies.

---

## 2.3 School-Year Context

Every new top-level Portia work context should be associated with one school year.

The authoritative initial source should be the owning class’s Core metadata:

```text
classes/<class_id>/class.json
```

Portia should copy the class `school_year` into its top-level work metadata when that work is created.

That stored value should remain stable even if:

* the active workspace school year later changes;
* the class metadata is later corrected;
* or the Portia work becomes historical.

The active workspace school year may be used for:

* teacher-facing defaults;
* setup validation;
* and mismatch warnings.

It must not silently replace the school year recorded in the owning class metadata or in an existing Portia record.

For writable Portia v1 workflows, a selected class should have valid Core class metadata with a valid `school_year`.

A class without metadata may remain a valid Core class, but Portia should treat it as incomplete setup and require the teacher to establish the class year before creating new Portia work.

The school year is contextual metadata. It is not:

* a student identifier;
* a class identifier;
* a Portia record identifier;
* or a substitute for an owning class.

---

## 2.4 Class Identity

Every writable Portia v1 Event or Support Process must have exactly one owning Core class.

The owning class is identified by:

```text
class_id
```

That class determines:

* the canonical work root;
* the primary school-year context;
* the presumptive instructional context;
* the default roster presented by teacher-facing workflows;
* and the Core routing context when returned paper is involved.

The owning class does not determine every student or Event that may be referenced. Explicit cross-class participant and work references may point to other valid classes and rosters in the same teacher workspace without changing ownership.

Portia may also reference instructional records associated with the owning or contextual classes, but those references do not change canonical ownership.

### Unsupported Class Scopes

Portia v1 does not support an Event or Support Process canonically owned by:

* several classes simultaneously;
* no legitimate class;
* a school;
* a district;
* a hallway or cafeteria as an independent institutional context;
* or another teacher’s roster or workspace.

Portia must not handle such cases by:

* choosing an arbitrary class;
* creating a synthetic `schoolwide` class;
* duplicating the Event under several classes;
* or treating the workspace itself as the Event or Support Process owner.

Cross-class participants and cross-class work relationships do not create multi-class ownership. They are explicit references from one canonically owned work item.

When a requested workflow has no legitimate owning class, Portia should reject it clearly as outside the initial product scope.

A teacher may still describe relevant surrounding context in a Portia record, but contextual text does not create a new ownership scope.

---

## 2.5 Roster Identity

Every roster-qualified student reference uses a canonical Core roster:

```text
classes/<class_id>/roster.csv
```

Portia should not create or maintain a separate canonical roster.

The owning class’s roster is the default source presented when creating Event Participants or student-centered Support Processes. An explicit cross-class workflow may instead select a student from another valid Core roster in the same teacher workspace.

For each selected roster, the current roster is authoritative for:

* selecting students in new workflows;
* validating that the referenced `student_id` currently belongs to that roster;
* and displaying current roster information.

The roster itself has no independent `roster_id` or revision identifier in the current Core contract.

Portia should therefore identify each source roster through:

```text
class_id
+ canonical roster location
```

rather than by inventing a Portia-specific roster identifier.

### Roster Changes

Removing a student from the active roster must not invalidate historical Portia records.

A persisted Portia student relationship should remain readable even when the current roster:

* no longer contains the student;
* has corrected the student’s name;
* has changed optional fields;
* or is temporarily unavailable.

Portia should distinguish:

```text
historical recorded student reference
```

from:

```text
successful resolution against the current roster
```

Failure to resolve a historical reference against the current roster is a validation or display condition. It is not permission to delete, rewrite, or reassign the historical record.

---

## 2.6 Student Identity

For Portia v1, one student is identified by the composite reference:

```text
class_id + student_id
```

The `class_id` identifies the source roster. The `student_id` identifies the student within that roster. The source roster may be the owning class’s roster or another valid roster explicitly linked within the same teacher workspace.

A student reference should conceptually use a structure such as:

```json
{
  "class_id": "english10_p2",
  "student_id": "1001"
}
```

Portia may store a display snapshot alongside that reference:

```json
{
  "display_name": "Jane Doe"
}
```

The display snapshot exists only to preserve historical readability. It is not identity and must not override the durable reference.

### Roster-Qualified Identity Without Automatic Merging

Portia v1 must not assume that matching `student_id` values in two classes identify the same person.

It also must not attempt to determine identity by comparing:

* first and last names;
* preferred names;
* initials;
* email addresses stored in optional roster columns;
* or other approximate metadata.

The same real-world student enrolled in two of the teacher’s classes is represented through two separate roster-qualified references:

```text
english10_p2 + 1001
journalism_p6 + 1001
```

Portia v1 does not merge those references into one canonical student identity. A teacher may explicitly select several roster-qualified references for a derived cross-class history or report, but the underlying identities remain distinct.

This is a deliberate product boundary, not a temporary identity-resolution heuristic.

### No Automatic Cross-Year Person Identity

A student appearing in a later school year is not automatically the same Portia identity, even when the same `student_id` is reused.

When a Support Process continues into a later class or school-year context, cross-year continuity is represented through an explicit predecessor-successor relationship. The successor Support Process owns the canonical `successor_of` relationship to the prior Support Process.

Continuity must not be created through an assumed permanent student dossier.

---

## 2.7 Local Operator Attribution

Portia v1 is operated by one teacher in a local workspace.

It does not have authenticated user accounts.

Portia may allow the teacher to configure a local display identity such as:

```text
Stephen Severino
Teacher
```

That value may be stored as record attribution, but it must not be described as:

* an authenticated identity;
* a verified electronic signature;
* an authorization decision;
* or proof that no other person used the local computer.

Portia should distinguish:

```text
recorded author label
```

from:

```text
authenticated user identity
```

The initial product provides the former only.

Every teacher-entered record should still contain explicit provenance such as:

* author or recorder label;
* creation timestamp;
* last-updated timestamp;
* and source type.

This supports local accountability without overstating the security model.

---

## 2.8 Statement and Account Attribution

The person who enters an Account is not necessarily the person whose perspective it represents.

Portia must separate:

* the local operator who entered the record;
* the attributed source of the Account;
* the student or students the Account concerns;
* and the owning class.

An Account or statement may be attributed to:

* the teacher;
* a roster student from the owning class;
* a roster student explicitly linked from another class taught by the teacher;
* a recurring non-roster Actor;
* an incidental or unidentified descriptive source;
* or another externally identified source.

A roster student serving as a source uses a roster-qualified `class_id + student_id` reference rather than an Actor record or name-only label.

A recurring non-roster person may be represented through the teacher-local Portia Actor Directory. The work-specific record stores the `actor_id`, a historical display snapshot, and the person’s contextual role in that Account or statement.

When no Actor record is appropriate, an incidental, unidentified, or one-time source may use controlled descriptive fields such as:

```text
source_type
source_role
source_display_label
```

For example:

```json
{
  "source_type": "family_member",
  "source_role": "parent",
  "source_display_label": "Student's mother"
}
```

Descriptive attribution and Actor metadata remain teacher-entered information rather than institutionally verified identity.

Unknown or withheld sources should be represented explicitly, for example:

```text
unknown
withheld
not_recorded
```

They must not be represented through a fabricated person identifier.

---

## 2.9 Portia Work Identity

Every Portia top-level work context must use the Core module-qualified identity:

```text
portia + class_id + work_id
```

The corresponding Core type is:

```text
ModuleWorkRef
```

The `work_id` must be:

* durable;
* opaque;
* nonsemantic;
* path-safe;
* stable after creation;
* and unique within the owning class’s Portia work collection.

It must not be:

* the student ID;
* the student name;
* an Event description;
* a behavior category;
* a support tier;
* or a date-dependent filename label.

Portia v1 uses work-kind-specific diagnostic prefixes with opaque random content:

```text
evt_<random identifier>
sup_<random identifier>
```

Section 3 defines an Event or Support Process as the semantic unit represented by one `work_id`. The persisted work kind remains authoritative; the prefix is diagnostic rather than sufficient proof of type.

The selected `work_id` meaning remains distinct from every child domain-record identifier.

---

## 2.10 Portia Domain-Record Identity

Each durable Portia domain record should receive its own opaque identifier.

Likely record types include:

* Event;
* Event Participant;
* Account;
* Observation;
* Positive Observation;
* Concern;
* Referral;
* Immediate Response;
* Support;
* Intervention;
* Follow-Up;
* Outcome;
* Communication;
* Reentry;
* Repair;
* Amendment;
* and Disagreement.

Representative identifiers might use diagnostic prefixes:

```text
evt_<random>
ep_<random>
acct_<random>
sup_<random>
int_<random>
fu_<random>
out_<random>
```

The prefix may help developers and teachers recognize a record type, but the remainder must be nonsemantic.

An identifier must not encode:

* `student_id`;
* student initials;
* class period;
* behavior category;
* allegation;
* determination;
* support status;
* or outcome.

Record IDs should remain stable through:

* correction;
* status changes;
* closure;
* reopening;
* amendment;
* and archival movement.

A correction should not create a new identity merely because the active representation changed.

Where a genuinely new record supersedes an earlier record, the relationship should be represented explicitly rather than by reusing the old identifier.

---

## 2.11 Event-Participant Identity

An Event and a student are separate records.

Their relationship should be represented through an Event Participant record with its own durable identity.

Conceptually:

```text
Event
  -> Event Participant
      -> roster-qualified student reference
```

An Event Participant may later hold or link to participant-specific information such as:

* the student’s relationship to the Event;
* participant-specific Accounts;
* classifications;
* responses;
* determinations;
* follow-up;
* and outcomes.

The Event Participant identifier is not the same as `student_id`.

This distinction prevents the Event itself from implying that:

* every participant performed the same action;
* every Account applies equally to every participant;
* every participant received the same response;
* or every participant had the same outcome.

An Event may involve several students from the owning roster and may explicitly link students from other valid rosters in the same teacher workspace.

Each Event Participant stores its own complete `class_id + student_id` reference. The participant’s source `class_id` may differ from the Event’s owning `class_id`.

A cross-class participant must be selected through an explicit teacher-reviewed workflow. Portia must not import that student into the owning class roster, merge roster identities automatically, or duplicate the Event beneath the participant’s class.

---

## 2.12 External Module References

Portia may reference records from Core or another Paper Data Suite module without taking ownership of them.

An external reference should identify at least:

```text
owning module
record type
durable record identifier
contract version
relationship purpose
```

Portia remains authoritative for why the external record is relevant to a Portia workflow.

The source module remains authoritative for the external record itself.

An external reference must not be represented through:

* copied sibling-module data;
* a filesystem path;
* a display title;
* or an inferred relationship based on matching names.

The detailed typed-reference contract remains a later issue, but Issue #4 must preserve this identity boundary.

---

## 2.13 PDS2 Route Identity

A Portia record receives a PDS2 route identity only when Portia generates a physical page expected to return through scanning.

The route identity is:

```text
module_id + class_id + work_id + route_id
```

It identifies the expected returned page route.

It does not identify:

* the student;
* the Event Participant;
* the Account author;
* the subject of the page;
* or the final evidence relationship.

Those relationships remain in Portia-owned records.

A route target should use a Portia `ModuleRecordRef`, such as:

```text
portia / reflection_page / <record_id>
portia / check_in_page / <record_id>
portia / support_data_page / <record_id>
```

PDS2 is therefore a separate paper-routing identity layer, not the foundation of Portia’s normal digital record identity.

---

## 2.14 Derived Views and Indexes Have No Independent Domain Identity

Class lists, student histories, open-work queues, reports, exports, and other projections are derived from canonical Portia records.

They must not create competing domain identities.

An index may store durable record IDs for lookup, but the index entry is not the Event, Support, or Outcome itself.

Deleting or rebuilding an index must not alter the identity of any canonical Portia record.

Exports may have artifact identifiers for provenance, but an exported file must not become the authoritative Portia record.

---

## 2.15 Identity Invariants

The following invariants govern Portia v1:

1. Every Event and Support Process has exactly one legitimate owning Core class.
2. Subordinate workflow records inherit the owning class of their parent work item.
3. Workspace-scoped Actor records are the sole Portia v1 exception to class-owned canonical workflow storage.
4. A Portia student reference is `class_id + student_id`, where `class_id` identifies the source roster.
5. A selectable student may come from the owning roster or another valid roster explicitly selected within the same teacher workspace.
6. Portia performs no automatic cross-roster student merging.
7. Cross-class participants do not create cross-class ownership.
8. Names are display fields, never durable identifiers.
9. Historical student references remain valid even when the active roster changes.
10. A local author label does not imply authenticated identity.
11. The person entering an Account is distinct from the Account’s attributed source.
12. Every durable Portia domain record has its own opaque identifier.
13. An Event Participant has its own identity distinct from the student and Event.
14. Portia `work_id` is distinct from student IDs and child-record IDs.
15. Recurring non-roster people may use Portia Actor identities; incidental people may remain descriptive.
16. External records remain identified and owned by their originating modules.
17. PDS2 identifies returned paper routes, not students or behavior records.
18. Indexes, reports, and exports do not become competing sources of truth.
19. Unsupported identity scopes are rejected rather than coerced.
20. Cross-year continuity uses canonical successor relationships rather than an assumed permanent student identity.
21. A future institutional model would require a deliberate whole-suite redesign and is not a compatibility requirement for Portia v1.

## 3. The Semantic Meaning of `work_id` in Portia

### Decision

In Portia, one `work_id` identifies one independently managed, explicitly typed, top-level Portia workflow object.

The effective Core work identity remains:

```text
portia + class_id + work_id
```

The `work_id` does not identify:

* all Portia data for a class;
* a student dossier;
* a school-year storage bucket;
* an index;
* or an arbitrary directory created only to satisfy Core’s path structure.

It identifies one coherent unit of Portia work that can be:

* created;
* loaded;
* validated;
* updated;
* linked to other work;
* closed;
* reported on;
* and, when applicable, associated with returned paper routes.

### Initial Work Kinds

Portia v1 should support two top-level work kinds:

```text
event
support_process
```

Additional work kinds may be introduced later only through an explicit contract change.

---

### Event Work

An Event work item represents one bounded occurrence, observation context, or documented interaction.

Examples include:

* a positive behavior observation;
* a classroom behavior concern;
* a specific incident;
* a student self-report tied to a particular occurrence;
* a referral initiated from one documented occurrence;
* or another time-bounded classroom Event.

An Event work item may contain or reference:

* Event Participants;
* Accounts;
* objective Observations;
* quotations;
* positive observations;
* Concerns;
* Referrals;
* Classifications;
* Responses;
* communications;
* participant-specific determinations, where appropriate;
* Follow-Ups;
* Outcomes;
* amendments;
* and statements of disagreement.

A multi-student Event remains one Event work item, including when explicitly linked participants come from several of the teacher’s rosters.

It must not be duplicated into a separate work root for each student or participating class.

Conceptually:

```text
Event work
├── shared Event context
├── Event Participant A
│   ├── participant-specific accounts
│   ├── responses
│   └── outcomes
├── Event Participant B
│   ├── participant-specific accounts
│   ├── responses
│   └── outcomes
└── shared or source-specific Accounts
```

The existence of several participants does not change the Event’s owning class.

Every roster-student participant must resolve through a valid `class_id + student_id` reference in the same teacher workspace. The source roster may differ from the Event’s owning class.

---

### Support-Process Work

A Support Process work item represents one ongoing, teacher-managed effort to provide, implement, review, or adjust support within one owning instructional class context.

Examples include:

* a classroom behavior-support plan;
* a recurring check-in process;
* a replacement-skill intervention;
* an environmental or instructional support;
* a self-monitoring process;
* a reentry or repair process;
* or another ongoing teacher-controlled intervention.

A Support Process may contain or reference:

* its roster-qualified target student;
* its stated purpose;
* strengths and protective factors;
* goals;
* planned Supports;
* Interventions;
* implementation records;
* fidelity notes;
* linked Events;
* communications;
* Follow-Ups;
* Outcomes;
* revisions;
* closure;
* and successor relationships.

One Support Process may reference several Event work items.

Likewise, one Event may be relevant to more than one Support Process when those relationships are explicitly documented.

The records remain separate:

```text
Event work
    -> typed relationship
Support Process work
```

Portia must not copy the Event into the Support Process directory or copy the Support Process into each linked Event.

---

### Why Events and Support Processes Are Separate Work Kinds

Events and Support Processes have different lifecycle boundaries.

An Event is normally bounded by:

* an occurrence;
* a date or time range;
* a context;
* and a defined set of participants.

A Support Process is normally bounded by:

* a support purpose;
* an effective period;
* implementation activity;
* review;
* and eventual continuation, revision, or closure.

Treating every Support Process as a child of one Event would fail when the support responds to:

* several Events;
* a developing pattern documented through several observations;
* a student request;
* or a proactive need not originating in one incident.

Treating every Event as a child of one Support Process would fail when:

* the Event is standalone;
* it is a positive observation;
* no intervention is warranted;
* or the Event predates the Support Process.

Both therefore require independent top-level work identity.

---

### `work_id` as the Root Domain Identifier

The `work_id` should also serve as the durable identifier of the top-level Portia object.

An Event work item may use an identifier such as:

```text
evt_<random identifier>
```

A Support Process work item may use:

```text
sup_<random identifier>
```

For example:

```text
portia / english10_p2 / evt_7f3a9c...
portia / english10_p2 / sup_16b85d...
```

This avoids creating two separate identifiers for the same top-level object, such as:

```text
work_id
+ separate event_id for the root Event
```

For an Event work item:

```text
work_id = event_id
```

For a Support Process work item:

```text
work_id = support_process_id
```

Child records retain their own identifiers.

For example:

```text
evt_7f3a9c...     Event and work identity
ep_42d810...      Event Participant
acct_91ca40...    Account
resp_b5a822...    Response
fu_89ae31...      Follow-Up
out_463c1d...     Outcome
```

The diagnostic prefix helps identify the expected work kind, but the identifier’s remaining content is nonsemantic.

The authoritative work record must also declare its work kind explicitly. Portia must not depend solely on the identifier prefix to determine record type.

---

### Work Metadata and Type Validation

Each Portia work root should contain one canonical root record or manifest that declares at least:

```text
schema version
record type
work kind
module ID
class ID
work ID
school year
created timestamp
updated timestamp
lifecycle status
```

Conceptually:

```json
{
  "schema_version": "1",
  "record_type": "portia_work",
  "work_kind": "event",
  "module_id": "portia",
  "class_id": "english10_p2",
  "work_id": "evt_7f3a9c...",
  "school_year": "2026-2027",
  "created_at": "2026-09-18T10:24:00-04:00",
  "updated_at": "2026-09-18T10:24:00-04:00",
  "status": "active"
}
```

Loading through a canonical Core work path must verify that the persisted values match the containing `ModuleWorkRef`.

For example, a record stored beneath:

```text
classes/english10_p2/modules/portia/work/evt_7f3a9c.../
```

must not claim:

```text
class_id = english10_p3
```

or:

```text
work_id = evt_different...
```

The containing directory is not trusted by itself. The persisted identity and requested identity must agree.

---

### Work Creation

Creating a Portia work item should require:

* one valid owning Core class;
* valid class metadata;
* one school year;
* one supported work kind;
* one newly generated opaque `work_id`;
* and one valid initial root record.

Creation must be exclusive.

Portia must not overwrite an existing work root or silently reuse an existing `work_id`, even when the existing contents appear identical.

A work item should not become visible as valid merely because its directory exists. Creation should use staged or atomic behavior so that partial writes are not mistaken for complete work.

---

### Work Relationships

Relationships among Portia work items must use durable class-qualified references and one canonical relationship record.

For example, a Support Process may own a relationship to a relevant Event:

```json
{
  "record_kind": "work_relationship",
  "record_id": "rel_72a1...",
  "source": {
    "module_id": "portia",
    "class_id": "english10_p2",
    "work_id": "sup_16b85d...",
    "work_kind": "support_process"
  },
  "target": {
    "module_id": "portia",
    "class_id": "english10_p5",
    "work_id": "evt_7f3a9c...",
    "work_kind": "event"
  },
  "relationship": "informed_by_event",
  "status": "active"
}
```

The relationship is stored with the work item that creates, manages, or gives meaning to it. In this example, the Support Process owns the canonical link. The Event’s `linked Support Processes` display is a derived reverse view.

Portia must not require both work roots to contain independently editable copies of the same relationship.

---

### Lifecycle

A `work_id` remains stable throughout the work item’s lifecycle.

Possible lifecycle states may later include:

```text
active
closed
superseded
cancelled
invalidated
archived
```

Changing lifecycle state must not change the `work_id`.

Correcting an Event, revising a Support Process, adding an amendment, or recording a new Outcome must not create a new work identity unless the teacher is genuinely creating a separate top-level work item.

When a Support Process continues into a new class or school-year context, Portia creates a new successor work item. The successor owns the canonical relationship to the prior work item:

```text
new Support Process -> successor_of -> prior Support Process
```

The prior work item remains historically identifiable and need not be modified. A `succeeded_by` display from the prior work item is derived from the successor’s canonical relationship.

---

### Excluded Meanings

Portia `work_id` does not mean:

#### One class-year Portia collection

A class-year collection would provide little semantic distinction beyond `module_id + class_id` and would turn the work root into a broad storage bucket.

#### One student

Using one work root per student would encourage a permanent dossier, complicate multi-student Events, and combine unrelated workflows.

#### One directory per child record

Accounts, Follow-Ups, Outcomes, and other subordinate records receive their own record IDs but do not each require a separate Core `ModuleWorkRef`.

#### One printed document

A printed document or returned page is an artifact associated with an existing Event or Support Process. It is not ordinarily the top-level work identity.

#### One behavior category

Categories are classifications or descriptive metadata. They are mutable and must not determine storage identity.

---

### Work-ID Invariants

1. One `work_id` identifies one top-level typed Portia workflow object.
2. Portia v1 supports Event and Support Process work kinds.
3. The `work_id` is also the durable ID of the root Event or Support Process.
4. The work kind is declared explicitly in persisted data.
5. Identifier prefixes are diagnostic, not the sole source of type truth.
6. Every work item has exactly one owning Core class.
7. Every work item has one school-year context.
8. An Event may involve several roster-qualified students from one or more of the teacher’s classes without changing ownership.
9. A Support Process may reference several Event work items.
10. Events and Support Processes remain independently authoritative records.
11. Child records receive their own identifiers but remain within the parent work context.
12. A work item is not a class bucket or student dossier.
13. A `work_id` remains stable through correction, lifecycle changes, and archival movement.
14. Work relationships use durable references rather than copied records.
15. Unsupported work kinds are rejected rather than interpreted approximately.

## 4. Canonical Portia Workspace Layout

### Decision

Portia v1 will use Core’s existing class-qualified module work layout.

Each Event or Support Process receives one canonical work root:

```text
classes/<class_id>/modules/portia/work/<work_id>/
```

The `work_id` identifies the root Event or Support Process, as established in Section 3.

All canonical records belonging directly to that workflow are stored beneath its work root. Canonical Events, Support Processes, and their subordinate workflow records remain class-scoped. Portia maintains one limited workspace-scoped Actor Directory solely for reusable non-roster people who may participate across several of the teacher’s classes. Portia does not maintain a workspace-level Event store, Support store, or student dossier directory.

---

## 4.1 Work Collection

The Portia work collection for one class is:

```text
classes/<class_id>/modules/portia/work/
```

Its immediate child directories are independently managed Portia work items:

```text
classes/<class_id>/modules/portia/work/
  evt_<opaque-id>/
  evt_<opaque-id>/
  sup_<opaque-id>/
  sup_<opaque-id>/
```

A directory beginning with `evt_` is expected to contain an Event work item.

A directory beginning with `sup_` is expected to contain a Support Process work item.

The prefix is diagnostic only. The authoritative `work_kind` is declared inside the persisted root record.

Portia must not interpret a directory as valid work merely because its name has a recognized prefix.

---

## 4.2 Common Work-Root Layout

Every Portia work root uses this common structure:

```text
classes/<class_id>/modules/portia/work/<work_id>/
  work.json
  records/
  attachments/
  pages/
  routes/
  history/
  derived/
  exports/
```

Not every directory must exist immediately. Directories should be created only when required by an explicit workflow.

The only file required for a valid newly created work item is:

```text
work.json
```

---

## 4.3 `work.json`

`work.json` is the canonical root record for the Event or Support Process.

It contains:

* the complete work identity;
* the work kind;
* the owning class;
* the school year;
* lifecycle state;
* creation and update timestamps;
* and the root domain data for the Event or Support Process.

Conceptually:

```json
{
  "schema_version": "1",
  "record_type": "portia_work",
  "work_kind": "event",
  "module_id": "portia",
  "class_id": "english10_p2",
  "work_id": "evt_7f3a9c...",
  "school_year": "2026-2027",
  "status": "active",
  "created_at": "2026-09-18T10:24:00-04:00",
  "updated_at": "2026-09-18T10:24:00-04:00"
}
```

The exact Event- and Support-Process-specific fields will be defined in the Portia domain-model issue.

Portia must validate that the persisted identity matches the containing Core path.

For example, a file stored beneath:

```text
classes/english10_p2/modules/portia/work/evt_7f3a9c.../
```

must declare:

```text
module_id = portia
class_id = english10_p2
work_id = evt_7f3a9c...
work_kind = event
```

A mismatch makes the work item invalid.

The containing directory alone is not authoritative.

---

## 4.4 Canonical Child Records

Canonical subordinate records are stored beneath:

```text
records/<record_kind>/<record_id>.json
```

For example:

```text
records/

    ep_42d810....json
  account/
    acct_91ca40....json
  observation/
    obs_72b941....json
  classification/
    cls_0e97b2....json
  response/
    resp_b5a822....json
  implementation/
    impl_63f914....json
  follow_up/
    fu_89ae31....json
  outcome/
    out_463c1d....json
  communication/
    comm_4784ab....json
  amendment/
    amd_45fd90....json
  work_relationship/
    rel_b312ce....json
```

The available record kinds depend on the parent work kind.

An Event may contain records such as:

* Event Participants;
* Accounts;
* Observations;
* Classifications;
* immediate Responses;
* Follow-Ups;
* Outcomes;
* Communications;
* and Amendments.

A Support Process may contain records such as:

* planned actions;
* implementation records;
* fidelity records;
* Follow-Ups;
* Outcomes;
* Communications;
* revisions;
* and work relationships.

The final record-kind vocabulary and schemas remain domain-model decisions.

### Child-Record Rules

Each canonical child record must:

* have one opaque `record_id`;
* declare its `record_kind`;
* identify its parent `class_id` and `work_id`;
* use a filename matching its `record_id`;
* and have one authoritative file.

For example:

```text
records/account/acct_91ca40....json
```

must contain:

```text
record_kind = account
record_id = acct_91ca40...
class_id = <owning class>
work_id = <owning Portia work>
```

Portia must reject mismatches rather than infer identity from the path.

A child record must not be copied into another work root merely to make it visible there.

A multi-student Event, including an Event with explicit cross-class participant links, may use:

  work.json

  records/
    event_participant/
      ep_a12c....json
      ep_b94e....json

    account/
      acct_01a7....json
      acct_876d....json

    observation/

Links to relevant Event work items are represented through `work_relationship` records.

The linked Events remain authoritative in their own work roots.

## 4.7 Work Relationships

Relationships between Events and Support Processes are represented through durable work references.

A relationship record should identify:

```text
source class ID
source work ID
source work kind
target class ID
target work ID
target work kind
relationship type
```

For Portia v1, both work items must belong to valid classes in the same teacher workspace. Cross-class relationships are permitted only through explicit class-qualified references and teacher-reviewed linking. Each work item retains its own owning class and canonical location.

Example:

```json
{
  "record_kind": "work_relationship",
  "record_id": "rel_72a1...",
  "source": {
    "module_id": "portia",
    "class_id": "english10_p2",
    "work_id": "sup_16b85d...",
    "work_kind": "support_process"
  },
  "target": {
    "module_id": "portia",
    "class_id": "english10_p5",
    "work_id": "evt_7f3a9c...",
    "work_kind": "event"
  },
  "relationship": "supported_by_event"
}
```

One direction should be canonical.

For example, the Support Process may canonically store its links to Events. Event screens may derive reverse links by scanning relationship records.

Portia must not require independently editable relationship copies in both work roots.

---

## 4.8 Attachments

Supporting files may be stored beneath:

```text
attachments/<attachment_id>/
```

For example:

```text
attachments/
  att_512c.../
    attachment.json
    content.pdf
```

`attachment.json` is the canonical metadata record.

It should identify:

* the attachment ID;
* parent class and work;
* media type;
* original filename, when retained;
* creation or import timestamp;
* provenance;
* and applicable privacy classification.

The filesystem path must use only the opaque attachment ID.

Student names, Event descriptions, behavior categories, and other sensitive information must not appear in attachment directory names or filenames.

The original filename may be retained inside `attachment.json` for teacher display, but it is not identity.

---

## 4.9 Portia Page Records and Core Routes

Portia-generated pages expected to return through scanning use two distinct locations:

```text
pages/<page_record_id>.json
routes/<route_id>.json
```

For example:

```text
pages/
  pg_72a8....json

routes/
  rt_0123456789abcdef0123456789abcdef.json
```

Portia owns the page record beneath `pages/`.

Core owns the route-registration schema and persistence behavior beneath `routes/`.

The Portia page record contains the page’s semantic meaning, such as:

* page type;
* expected author or subject relationship;
* template version;
* related Event Participant;
* or intended workflow.

The Core route registration connects the PDS2 locator to the typed Portia page record.

The route file must be persisted successfully before the page is rendered.

Digital-only Portia work does not require either directory.

---

## 4.10 History Records

Portia must preserve append-only change history separately from the current canonical representation.

History records are stored beneath:

```text
history/<change_id>.json
```

For example:

```text
history/
  chg_0a73....json
  chg_918c....json
```

A history record may identify:

* the changed record;
* the operation type;
* the prior version or prior-value summary;
* the resulting version or resulting-value summary;
* the recorded author label;
* the timestamp;
* and the reason for change, when supplied.

History files are immutable after creation.

Updating a current canonical record should therefore follow this conceptual sequence:

```text
validate proposed change
→ prepare immutable history entry
→ safely replace current canonical record
→ preserve the history entry
```

The detailed transaction and recovery algorithm remains an implementation decision.

Portia must not silently rewrite or delete an existing history entry.

---

## 4.11 Derived Files

Rebuildable work-specific projections may be stored beneath:

```text
derived/
```

Examples include:

```text
derived/
  timeline.json
  summary.json
  participant_index.json
```

Derived files are not authoritative.

They may be:

* deleted;
* rebuilt;
* replaced;
* or ignored when stale.

Every derived file should identify enough source state to determine whether it is current, such as:

* generation timestamp;
* schema version;
* and source-record update markers.

A damaged or missing derived file must not prevent the canonical work from loading.

Portia must always be capable of rebuilding derived data from:

```text
work.json
records/
history/
```

---

## 4.12 Class-Wide Discovery and Indexing

Portia v1 does not require a persistent authoritative class-wide index.

To discover Portia work for a class, Portia should:

1. locate the class’s Portia work collection;
2. enumerate its immediate work directories;
3. load and validate each `work.json`;
4. reject or report invalid work roots;
5. build the required indexes in memory.

This is appropriate for the expected scale of one teacher’s classes and avoids introducing another source of truth.

In-memory indexes may include:

* work by kind;
* Events by date;
* work by student reference;
* active Support Processes;
* open Follow-Ups;
* and work by lifecycle status.

A future persisted cache may be added for performance, but it must be:

* explicitly nonauthoritative;
* fully rebuildable;
* safe to delete;
* and validated against canonical work records.

---

## 4.13 Exports

Generated teacher-facing files may be stored beneath:

```text
exports/
```

Examples include:

```text
exports/
  event_summary.pdf
  support_review.pdf
  student_projection.pdf
```

Exports are artifacts, not canonical records.

An export may be regenerated from the current canonical work.

Deleting an export must not delete or alter:

* the Event;
* Support Process;
* child records;
* history;
* or page-route registrations.

Export filenames should avoid unnecessary sensitive information. Student names should not be required in filenames.

Where an export requires a durable provenance record, that provenance should be stored as a canonical Portia record rather than inferred from the generated filename.

---

## 4.14 Work Creation and Staging

Creating a work item must not expose a partially written directory as valid Portia work.

A suitable creation process is:

```text
generate and validate work ID
→ create staged directory
→ write and synchronize work.json
→ atomically or safely rename staged directory to the canonical work root
```

A staged directory might temporarily use a location such as:

```text
work/.staging/<work_id>.<operation-id>/
```

The final canonical location remains:

```text
work/<work_id>/
```

Portia must create work exclusively.

If the final work root already exists, creation fails. Portia must not overwrite, merge with, or reinterpret the existing directory.

Abandoned staging directories may be reported and cleaned through an explicit recovery workflow. They must not appear in normal Portia work listings.

---

## 4.15 Canonical File Updates

Updates to an existing canonical JSON file should use staged file replacement:

```text
validate complete replacement record
→ write temporary file beside target
→ flush and synchronize
→ replace target
```

Portia should not update canonical JSON by editing bytes in place.

A failure before replacement should leave the previous canonical file intact whenever practical.

Every loaded record must be validated after parsing. A file is not trusted merely because it appears at the expected path.

---

## 4.16 Invalid and Incomplete Work Roots

A work directory is invalid or incomplete when, for example:

* `work.json` is missing;
* `work.json` is malformed;
* persisted identity does not match the path;
* the work kind is unsupported;
* required root fields are missing;
* a child filename does not match its record ID;
* or a child record claims another class or work owner.

Invalid work must not be silently omitted from all teacher awareness.

Portia should distinguish between:

* valid work;
* incomplete staged work;
* invalid canonical work;
* and unsupported future-version work.

Teacher-facing recovery behavior will be defined later, but ordinary workflows must not write into an invalid work root.

---

## 4.17 Deletion and Lifecycle

Portia v1 should prefer lifecycle transitions over physical deletion of canonical work.

Closing, cancelling, invalidating, or superseding a work item changes its lifecycle state but does not change its path or `work_id`.

Physical deletion, destruction eligibility, archival movement, and legal or records holds remain future Sunset and retention decisions.

Portia must not implement ordinary “delete” behavior as recursive removal of a work root without a separately accepted policy.

---

## 4.18 Layout Invariants

1. Every Event or Support Process has one canonical Core work root.
2. `work.json` is the authoritative root record.
3. The root path and persisted identity must agree exactly.
4. Child records use one file per durable record ID.
5. Child records identify their owning class and work.
6. Multi-student Events, including Events with cross-class participant links, remain one work root.
7. Supports and Events reference one another rather than copying records.
8. Student names and sensitive semantics do not appear in canonical path components.
9. Portia page records and Core route registrations remain separate.
10. History records are immutable after creation.
11. Derived files and exports are not sources of truth.
12. Portia v1 builds class-wide indexes from canonical work rather than relying on an authoritative index file.
13. Partial creation must not appear as valid work.
14. Canonical JSON updates use complete validated replacement rather than in-place editing.
15. Lifecycle changes do not change `work_id` or canonical location.
16. Invalid work roots are reported and are not silently rewritten.

## 5. Event Ownership and Cross-Class Participants

### Decision

Every Portia Event has exactly one owning Core class.

The owning class establishes:

* the Event’s canonical storage location;
* its Core `ModuleWorkRef`;
* its PDS2 routing context, when paper routing applies;
* its primary school-year context;
* and the instructional context in which the teacher was acting.

The owning class does not necessarily contain every student participant.

A Portia Event may link students from other valid Core rosters in the same teacher’s workspace without duplicating the Event.

---

## 5.1 Temporal and Instructional Ownership

Class ownership should reflect the real instructional context of the Event rather than an arbitrary storage choice.

Because a teacher generally teaches one scheduled class at a time, the Event’s date, time, and instructional activity ordinarily make the owning class apparent.

The presumptive rule is:

> When an Event occurs during a scheduled class period, the class being taught at that time owns the Event.

For example, an Event occurring during English 10 Period 2 belongs to that English 10 class even when one participant is linked from another roster.

The presence of a cross-class participant does not transfer or divide ownership.

Other legitimate owning contexts may include:

* a class field trip;
* a scheduled class assembly;
* an online meeting for the class;
* a class-specific extracurricular activity supervised by the teacher;
* or another activity clearly conducted on behalf of one class.

Canonical storage follows this real-world temporal and instructional ownership.

It does not create that ownership artificially.

---

## 5.2 Selecting the Owning Class

For a PDS2-routed paper artifact, the owning class is the `class_id` already encoded in the validated route.

For a digitally created Event, the teacher must select or confirm the owning class.

Portia may use the Event’s timestamp and an available teacher schedule to:

* suggest the class being taught;
* preselect the likely class;
* warn when the selected class conflicts with the schedule;
* or explain that no scheduled class matches.

Schedule-based selection remains assistive.

The teacher retains final responsibility for confirming the correct instructional context.

Portia must not silently change the owning class after Event creation merely because schedule information changes.

---

## 5.3 Cross-Class Student Participants

A student from another class taught by the same teacher may be linked as an Event Participant.

Every student reference remains roster-qualified:

```json
{
  "class_id": "english10_p5",
  "student_id": "2047"
}
```

A cross-class Event may therefore contain participants such as:

```text
Owning class: english10_p2

Participants:
- english10_p2 + 1001
- english10_p2 + 1014
- english10_p5 + 2047
```

The Event remains stored only beneath the owning class:

```text
classes/english10_p2/modules/portia/work/<event_id>/
```

Portia must not:

* duplicate the Event beneath `english10_p5`;
* create a second Event for the linked student;
* move the Event because another roster contributes a participant;
* or copy participant-specific records into another class’s work collection.

The cross-class relationship is represented through Event Participant records or equivalent canonical manifest data.

---

## 5.4 Cross-Class Linking Workflow

Portia should expose an explicit workflow such as:

```text
Link student from another class
```

The workflow should be available through both:

* the teacher-facing menu;
* and the CLI.

The workflow should require the teacher to:

1. select another valid Core class;
2. select a student from that class’s valid roster;
3. choose or confirm the participant relationship;
4. review the owning class and linked roster identity;
5. confirm the link.

Portia must validate that:

* the referenced class exists;
* its roster is valid;
* the referenced `student_id` exists in that roster;
* the participant reference includes both `class_id` and `student_id`;
* and the participant is not already linked through the same roster-qualified identity.

Portia must not link students automatically because names or student IDs appear similar across rosters.

---

## 5.5 Participant Identity and Event Ownership Remain Distinct

The following identities must not be conflated:

```text
Event owning class
Participant roster class
Participant student ID
Event work ID
Event Participant ID
```

A participant’s roster class identifies the source of that student identity.

It does not imply that the participant’s class owns the Event.

Likewise, the Event’s owning class does not imply that every participant appears on that roster.

An Event Participant should therefore contain or reference:

```text
event participant ID
owning Event work ID
participant class ID
participant student ID
participant relationship
```

The Event Participant ID remains distinct from the student reference.

---

## 5.6 People Not Present on a Teacher Roster

A person who does not appear in any valid roster available to the teacher cannot be represented as a roster-qualified Portia student participant.

A recurring non-roster person may instead be referenced through a deliberate Portia Actor record. An incidental, unidentified, or one-time person may remain descriptive. These representations may serve as:

* a contextual actor;
* an attributed Account source;
* a family member;
* a staff member;
* an outside student;
* an unknown person;
* or another explicitly labeled external role.

Portia must not create:

* a fabricated student;
* a temporary roster entry;
* a synthetic class;
* or an invented `student_id`

merely to represent that person.

If the person later appears on one of the teacher’s valid rosters, linking that roster identity to the earlier contextual record would require an explicit reviewed operation. Portia must not perform that conversion automatically.

---

## 5.7 Events Without an Obvious Scheduled Class

Some Events may occur:

* during lunch;
* before or after school;
* between periods;
* during an unscheduled supervision duty;
* or in another context without an immediately obvious class owner.

In these cases, Portia should require the teacher to select the most legitimate instructional class context.

The selected class should be one for which the teacher can honestly state that the Event belongs to that teacher-student instructional relationship.

Portia must not describe this choice as arbitrary storage assignment.

If no class provides an honest instructional context, the Event falls outside the normal Portia v1 model.

Portia may reject the workflow or require an explicitly defined exceptional process rather than creating a false class relationship.

The exceptional process, if needed, should be designed separately and must not introduce a synthetic `schoolwide` class.

---

## 5.8 Ownership Stability

Once created, an Event’s owning class should remain stable.

Adding or removing participants does not change ownership.

Correcting a participant’s roster reference does not change ownership.

Linking a student from another class does not change ownership.

If an Event was genuinely created under the wrong owning class, Portia should not silently edit its `class_id`, because that field controls:

* canonical storage;
* work identity;
* route identity;
* and historical context.

Correction may require a deliberate migration or superseding operation that preserves the original provenance.

The detailed correction workflow remains an implementation decision.

---

## 5.9 Cross-Class Boundaries

Portia v1 permits:

* one owning class;
* participants from several rosters belonging to the same teacher;
* same-workspace roster lookup;
* and explicit teacher-reviewed cross-class linking.

Portia v1 does not provide:

* cross-teacher roster lookup;
* schoolwide student search;
* district identity resolution;
* institution-wide Event ownership;
* shared administrative cases;
* or multi-user access to the same Event.

Those capabilities would require a broader Paper Data Suite redesign and are not compatibility requirements for the teacher-local model.

---

## 5.10 Ownership and Participant Invariants

1. Every Event has exactly one owning Core class.
2. The owning class normally reflects the Event’s temporal and instructional context.
3. A scheduled class occurring at the Event time is the presumptive owner.
4. Routed paper uses the class encoded in its validated PDS2 route.
5. Digitally created Events require teacher confirmation of the owning class.
6. Schedule information may assist selection but does not replace teacher judgment.
7. An Event may include students from other classes taught by the same teacher.
8. Every student participant uses a complete `class_id + student_id` reference.
9. Cross-class participants do not change the Event’s owning class.
10. One Event has one canonical work root and is never duplicated across classes.
11. Cross-class linking is an explicit, validated, teacher-reviewed operation.
12. Matching IDs or names across rosters never create automatic links.
13. People absent from all teacher rosters use a deliberate Actor reference when recurring or a descriptive contextual/source representation when incidental or unidentified.
14. Portia does not fabricate classes or students to accommodate unsupported identities.
15. Changing an Event’s owning class requires an explicit provenance-preserving correction process.
16. Cross-teacher and institution-wide identity remain outside Portia v1.

## 6. Canonical Relationships and Derived Views

### Decision

Portia will store one canonical record for each relationship.

Reverse links, student histories, timelines, dashboards, and other navigation views will be derived from canonical work and relationship records.

Portia will not maintain independently editable copies of the same relationship in several locations.

---

## 6.1 Canonical Relationship Ownership

A relationship record should normally be stored with the work item that creates, manages, or gives the relationship its meaning.

For example, when a Support Process is created in response to several Events, the Support Process owns the canonical links:

```text
Support Process S
  -> Event A
  -> Event B
```

The relationship records are stored beneath the Support Process work root:

```text
classes/<class_id>/modules/portia/work/<support_id>/
  records/
    work_relationship/
      <relationship_id>.json
```

Event A and Event B do not store editable reverse copies.

When either Event is viewed, Portia derives its linked Support Processes by reading canonical relationship records.

---

## 6.2 Relationship Records

Each relationship record should have its own opaque identifier and identify:

* the source work;
* the target work;
* both work kinds;
* the relationship type;
* creation provenance;
* lifecycle status;
* and applicable timestamps.

Conceptually:

```json
{
  "record_kind": "work_relationship",
  "record_id": "rel_72a1...",
  "source": {
    "module_id": "portia",
    "class_id": "english10_p2",
    "work_id": "sup_16b85d...",
    "work_kind": "support_process"
  },
  "target": {
    "module_id": "portia",
    "class_id": "english10_p2",
    "work_id": "evt_7f3a9c...",
    "work_kind": "event"
  },
  "relationship": "supported_by_event",
  "status": "active"
}
```

The exact relationship vocabulary will be defined with the domain model.

Portia must not infer a relationship solely because two records:

* concern the same student;
* occurred near the same time;
* use similar classifications;
* or contain similar text.

Relationships are explicit teacher-reviewed records.

---

## 6.3 Reverse Links

Reverse links are views, not separate canonical records.

For example, if a Support Process stores:

```text
Support Process S -> Event A
```

the Event screen may display:

```text
Linked Support Process: S
```

That reverse view is calculated from the canonical relationship.

Removing, invalidating, or superseding the canonical relationship must automatically affect the next generated reverse view.

Portia must not require the teacher to edit both sides of one relationship.

---

## 6.4 Student Histories

Portia will not create one authoritative student-history file or student dossier.

A student history is generated by finding canonical records associated with an exact roster-qualified student reference:

```text
class_id + student_id
```

The generated history may include:

* Event participation;
* positive observations;
* Accounts attributed to or concerning the student;
* Responses;
* Support Processes;
* implementation records;
* Follow-Ups;
* Outcomes;
* communications;
* amendments;
* and lifecycle changes.

Cross-class references remain distinct.

When a student appears through two roster identities, Portia does not merge those histories automatically.

A student-history view may allow the teacher to include explicitly selected cross-class participant references, but the underlying identities remain roster-qualified and separate.

---

## 6.5 Timelines and Dashboards

Portia may derive views such as:

* Events by date;
* active Support Processes;
* overdue Follow-Ups;
* recent positive observations;
* work by student;
* work by status;
* linked Events and Supports;
* and class-level activity timelines.

These views are calculated from canonical:

```text
work.json
records/
history/
```

A dashboard entry is not an Event, Support Process, Follow-Up, or Outcome.

Editing a dashboard entry must invoke an operation on the underlying canonical record rather than modifying a separate dashboard copy.

---

## 6.6 Persisted Indexes and Caches

Portia v1 may build required indexes in memory by scanning validated work roots.

A future implementation may persist indexes for performance.

Any persisted index must be:

* explicitly nonauthoritative;
* fully rebuildable;
* safe to delete;
* versioned;
* and traceable to the canonical records from which it was generated.

Possible locations include:

```text
classes/<class_id>/modules/portia/derived/
```

or work-specific:

```text
classes/<class_id>/modules/portia/work/<work_id>/derived/
```

A stale, missing, or corrupt index must not alter canonical data or prevent direct loading of valid work.

Portia should rebuild or ignore the index rather than treating it as a competing source of truth.

---

## 6.7 Relationship Validation

When loading a relationship, Portia must validate that:

* the source work exists;
* the target reference is structurally valid;
* the referenced work kind matches the target work;
* both records are within Portia’s permitted teacher-local scope;
* the relationship type is valid for the two work kinds;
* and the relationship record belongs to its containing source work.

A missing target does not authorize Portia to fabricate or recreate it.

Portia should report the relationship as unresolved, broken, or historical according to later lifecycle rules.

---

## 6.8 Relationship Lifecycle

Removing an active relationship should normally mean changing its lifecycle state or recording a superseding amendment rather than erasing its history.

Possible states may include:

```text
active
inactive
superseded
invalidated
```

The relationship ID remains stable.

Historical reports may show that a relationship previously existed even when it is no longer active.

The exact transition rules will be defined with the domain and correction models.

---

## 6.9 Canonical Cross-Year Successor Relationship

When a Support Process continues into a new class or school-year context, the new Support Process owns the canonical `successor_of` relationship to the prior Support Process.

```text
new Support Process -> successor_of -> prior Support Process
```

The relationship record is stored beneath the successor work root because the successor initiates and manages the continuation.

The prior Support Process remains unchanged and historically complete. A `succeeded_by` link displayed from the prior Support Process is a derived reverse view, not a second canonical relationship.

The successor receives a new `work_id`, owning class, school-year context, lifecycle, and history. The predecessor keeps its original identity, storage, and provenance.

---

## 6.10 Canonical and Derived Invariants

1. Every relationship has one canonical record.
2. The work item that creates or manages the relationship normally owns that record.
3. Reverse links are derived rather than independently stored.
4. Student histories are generated views, not permanent dossiers.
5. Dashboards and timelines are not sources of truth.
6. Correcting a canonical record affects all subsequently generated views.
7. Persisted indexes, if introduced, are disposable and rebuildable.
8. A missing index does not invalidate canonical work.
9. Relationships are explicit and are not inferred from similarity or proximity.
10. Broken references are reported rather than silently repaired.
11. Relationship lifecycle changes preserve the relationship’s durable identity and history.
12. A successor Support Process owns the canonical `successor_of` relationship to its predecessor; `succeeded_by` is derived.

## 7. Participant References and the Portia Actor Directory

### Decision

Portia will use two durable identity systems for people involved in its workflows:

1. **Roster students** use Core roster-qualified references:

   ```text
   class_id + student_id
   ```

2. **Recurring non-roster people** use opaque Portia-owned Actor references:

   ```text
   actor_id
   ```

Incidental, unidentified, or one-time people may still be represented descriptively without creating a durable Actor record.

This distinction allows Portia to reuse recurring parents, counselors, administrators, paraprofessionals, and other collaborators without creating a general schoolwide person registry.

---

## 7.1 Historical Student References

A persisted student reference must contain:

* the source `class_id`;
* the source `student_id`;
* and a nonauthoritative display snapshot.

Conceptually:

```json
{
  "student_ref": {
    "class_id": "english10_p2",
    "student_id": "1001"
  },
  "display_snapshot": {
    "first_name": "Jane",
    "last_name": "Doe",
    "preferred_name": null,
    "display_name": "Jane Doe"
  }
}
```

The roster-qualified reference is identity.

The display snapshot preserves historical readability when the current roster:

* changes the student’s name;
* changes a preferred name;
* removes the student;
* becomes unavailable;
* or no longer resolves the historical reference.

Portia may display both current and recorded information when they differ, but it must not silently rewrite historical snapshots merely because the active roster changed.

A display snapshot must never be used to locate, merge, or reassign a student record.

---

## 7.2 Recurring Non-Roster Actors

A recurring non-roster person may receive a durable Portia Actor record.

Examples include:

* parent or guardian;
* counselor;
* administrator;
* case manager;
* paraprofessional;
* school psychologist;
* social worker;
* nurse;
* coach;
* or another recurring collaborator.

Each Actor receives an opaque identifier such as:

```text
actr_<random identifier>
```

The identifier must not encode:

* the person’s name;
* job title;
* relationship to a student;
* school;
* or contact information.

An Actor record may contain:

```json
{
  "schema_version": "1",
  "record_type": "portia_actor",
  "actor_id": "actr_8f31...",
  "display_name": "Maria Smith",
  "actor_type": "family_member",
  "role_labels": [
    "parent"
  ],
  "contact_details": {},
  "status": "active",
  "created_at": "2026-09-18T10:24:00-04:00",
  "updated_at": "2026-09-18T10:24:00-04:00"
}
```

The final schema and controlled vocabularies will be defined in the domain-model issue.

---

## 7.3 Actor Directory Scope

The Actor Directory belongs to one teacher’s selected Paper Data Suite workspace.

It is not:

* a school directory;
* a district directory;
* an employee directory;
* a student-information system;
* an authentication system;
* or an institutionally authoritative identity service.

It contains only people the teacher deliberately creates for reuse in that teacher’s Portia workflows.

Because the same Actor may participate in work across several classes, Actor records are workspace-scoped rather than owned by one class.

---

## 7.4 Canonical Actor Storage

Portia will define a small module-owned workspace-level registry:

```text
<PDS workspace>/
  portia/
    actors/
      <actor_id>.json
    history/
      actors/
        <change_id>.json
```

For example:

```text
portia/
  actors/
    actr_8f31....json
    actr_22a9....json
```

This path is a Portia-owned convention.

It is not currently supplied by Core’s class-qualified `ModuleWorkRef` helpers.

Portia should still use Core workspace-root resolution and shared identifier validation, but it must provide and document its own safe path helpers for this registry.

The workspace-level registry is limited to reusable Actor identity. Canonical Events, Support Processes, participant links, Accounts, and other workflow records remain stored in their class-owned Portia work roots.

---

## 7.5 Actor Identity Versus Contextual Role

An Actor identifies the recurring person.

It does not permanently define every role that person may have in every workflow.

For example, one counselor may appear as:

* an Account source;
* a consulted collaborator;
* an Intervention participant;
* a Follow-Up assignee;
* or a Communication recipient.

Those contextual relationships belong in the relevant Event or Support Process records.

A work-specific reference may use:

```json
{
  "actor_ref": {
    "actor_id": "actr_8f31..."
  },
  "display_snapshot": {
    "display_name": "Maria Smith"
  },
  "relationship": "family_contact"
}
```

The `actor_id` identifies the person.

The `relationship` identifies that person’s role in this particular record.

The display snapshot preserves historical readability.

---

## 7.6 Relationships Between Actors and Students

Portia must not treat relationships such as the following as inherent parts of Actor identity:

```text
parent of
guardian of
counselor for
case manager for
administrator reviewing
```

Those relationships may change, may be limited to a particular student or period, and may require context.

They should be stored as explicit relationship data in the relevant Portia workflow or in a later dedicated relationship contract.

For example:

```json
{
  "actor_id": "actr_8f31...",
  "student_ref": {
    "class_id": "english10_p2",
    "student_id": "1001"
  },
  "relationship": "parent_or_guardian"
}
```

Portia must not infer such a relationship from:

* matching last names;
* email addresses;
* prior communications;
* or repeated appearances in related work.

---

## 7.7 Actor Creation and Selection Workflow

Teacher-facing workflows should allow the teacher to:

1. search active Actor records;
2. select an existing Actor;
3. create a new Actor when no appropriate record exists;
4. use a descriptive one-time actor without creating a durable record;
5. review the selected Actor before saving the workflow record.

CLI and menu workflows should make the distinction explicit:

```text
Select existing Actor
Create recurring Actor
Record one-time or unidentified person
```

Portia should not create Actor records automatically from free-text names.

Creation must be a deliberate teacher action.

---

## 7.8 Incidental and Unidentified People

A durable Actor record is not required for every person mentioned in Portia.

A person may remain descriptive when the person is:

* unidentified;
* anonymous;
* incidental;
* unlikely to recur;
* outside the teacher’s normal workflow;
* or deliberately not named.

Conceptually:

```json
{
  "source_type": "external_person",
  "source_role": "student_from_another_teacher",
  "display_label": "Unidentified student",
  "actor_id": null
}
```

Portia must not fabricate an Actor ID merely to satisfy a schema.

A descriptive person may later be linked to a newly created Actor only through an explicit reviewed correction or amendment.

---

## 7.9 Actor Updates and Historical Records

Actor display and contact information may change over time.

Updating the current Actor record must not rewrite the snapshots stored in historical Events, Accounts, Communications, or Supports.

Historical records may therefore show:

```text
Recorded display: Maria Smith
Current Actor display: Maria Rodriguez
```

The durable `actor_id` establishes continuity.

The snapshots establish what was displayed or known when the historical record was created.

Actor updates should use complete validated replacement and append-only history consistent with Portia’s general correction model.

---

## 7.10 Actor Lifecycle

Actor identity should remain stable after creation.

An Actor may later become:

```text
active
inactive
retired
merged
invalidated
```

The exact lifecycle vocabulary and transition rules remain domain-model decisions.

Marking an Actor inactive or retired must not invalidate historical references.

Portia should not physically delete an Actor record that is referenced by canonical work.

When duplicate Actor records are discovered, Portia must not silently combine them. A future reviewed merge process should:

* identify the surviving Actor;
* preserve the retired Actor identity;
* record the merge provenance;
* and allow historical references to remain explainable.

---

## 7.11 Contact Information

Contact information is optional and should be limited to what is useful for the teacher’s Portia workflows.

Possible fields may include:

* email address;
* phone number;
* preferred contact method;
* or a short teacher-entered note.

Portia should not require contact information merely to create an Actor.

Contact information is sensitive data and must not appear in:

* `actor_id`;
* directory names;
* filenames;
* route IDs;
* or exported filenames.

Reports should include contact details only when the report purpose explicitly requires them.

---

## 7.12 Actor Directory Discovery

Portia may discover Actor records by enumerating:

```text
<PDS workspace>/portia/actors/
```

Each filename must match the contained `actor_id`.

Portia should load and validate each record independently.

Actor search views may be indexed in memory by:

* display name;
* actor type;
* role label;
* status;
* or recent use.

A future persisted search index may be added, but it must remain derived, disposable, and rebuildable.

---

## 7.13 Identity Invariants

1. Roster students use `class_id + student_id`.
2. Historical student references include nonauthoritative display snapshots.
3. Roster changes do not rewrite historical student snapshots.
4. Recurring non-roster people may receive opaque Portia `actor_id` values.
5. Actor records are workspace-scoped and teacher-local.
6. The Actor Directory is not an institutional person registry.
7. Events and Support Processes remain class-owned despite referencing workspace-scoped Actors.
8. Actor identity is separate from the Actor’s role in a particular workflow.
9. Relationships between Actors and students are explicit and are not inferred.
10. Incidental or unidentified people may remain descriptive without an Actor record.
11. Portia never creates Actors automatically from free-text names.
12. Historical Actor references remain valid after Actor metadata or lifecycle changes.
13. Current Actor records and historical display snapshots may legitimately differ.
14. Referenced Actor records are not physically deleted through ordinary workflows.
15. Actor contact data never appears in identifiers or canonical path components.

1. **Class-owned workflow storage** for Events, Support Processes, and their subordinate records.

No other workspace-scoped canonical Portia record collections are required.

## 8.2 Authoritative Storage

The following are canonical:

* `portia/actors/<actor_id>.json`;
* Event `work.json`;
* Support Process `work.json`;
* records beneath `records/`;
* attachment metadata and retained attachment content;
* Portia page records;
* Core route registrations;
* and append-only history records.

Each canonical record has exactly one authoritative location.

A record must not be copied into another class, work root, student folder, timeline, or report merely to make it discoverable.

---

## 8.3 Derived Storage

The following are nonauthoritative:

* Actor search indexes;
* Event timelines;
* participant indexes;
* student histories;
* reverse relationship views;
* summaries;
* dashboards;
* reports;
* and exports.

Derived data may be regenerated from canonical records.

A missing, stale, or corrupt derived file must not invalidate otherwise valid Portia work.

---

## 8.4 Cross-Class References

A class-owned Event may contain Event Participants from other valid rosters in the teacher’s workspace.

A Support Process may also reference relevant Events owned by another class when:

* both classes exist in the same teacher workspace;
* all roster-qualified references are valid;
* the relationship is created explicitly;
* and the teacher confirms its relevance.

For example:

```text
Support Process:
classes/english10_p2/modules/portia/work/sup_16b85d.../

Linked Event:
classes/english10_p5/modules/portia/work/evt_7f3a9c.../
```

Neither work item changes ownership.

The relationship record contains the complete class-qualified identities of both work items.

Portia does not duplicate either work item into the other class.

---

## 8.5 Workspace Copy and Merge Boundary

Portia identifiers are durable within one teacher’s workspace.

Copying or moving the complete workspace preserves those identities because relative storage and record IDs remain intact.

Portia v1 does not support:

* merging two independently created workspaces;
* reconciling duplicate Actor IDs across workspaces;
* combining class records from separate teachers;
* or determining whether two workspace copies have diverged.

Those operations would require a broader suite-level identity and synchronization contract.

---

## 8.6 Storage Invariants

1. Events and Support Processes remain class-owned.
2. Reusable non-roster Actors remain workspace-scoped.
3. No canonical student dossier exists.
4. Every canonical record has one authoritative location.
5. Cross-class relationships use complete class-qualified references.
6. Cross-class linking never duplicates canonical work.
7. Derived indexes and views are disposable.
8. Absolute workspace paths are not persisted as identity.
9. Sensitive names and behavior descriptions do not appear in canonical path components.
10. Workspace merging is outside Portia v1.

## 9.2 Required Core Changes

No blocking `pds-core` change is required for Portia v1.

Portia can use existing Core behavior for:

* workspace-root resolution;
* class discovery;
* class metadata;
* school-year validation;
* roster loading and validation;
* shared identifier validation;
* class-qualified module work paths;
* safe work descendants;
* PDS2 routing;
* and route-registration persistence.

Portia owns the remaining domain and persistence behavior.

---

## 9.3 Portia-Owned Infrastructure

Portia must implement:

* Event and Support Process ID generation;
* child-record and Actor ID generation;
* work-root creation and validation;
* canonical record schemas;
* safe atomic writes;
* append-only history;
* cross-work reference validation;
* cross-class participant lookup;
* Actor Directory paths and persistence;
* derived indexes and histories;
* lifecycle behavior;
* and teacher-facing recovery diagnostics.

Portia must use Core’s validation and path contracts where they apply rather than reproducing them.

---

## 9.4 Workspace-Scoped Actor Path

The Actor Directory is a deliberate Portia-owned path convention:

```text
<PDS workspace>/portia/actors/
```

Core currently has no generic workspace-scoped module-root helper.

Portia may safely implement focused helpers such as:

```python
portia_workspace_dir(root)
actor_collection_dir(root)
actor_record_path(root, actor_id)
actor_history_dir(root)
```

These helpers must:

* accept the resolved Core workspace root;
* validate identity-bearing path components;
* perform no hidden filesystem creation;
* and reject unsafe paths rather than sanitize them.

---

## 9.5 Optional Future Core Standardization

A future Core enhancement could define a generic workspace-level module path such as:

```text
<PDS workspace>/modules/<module_id>/
```

or expose a helper such as:

```python
workspace_module_dir(root, module_id)
```

That change is not required for Portia v1.

It should be considered only if several Paper Data Suite modules independently require workspace-scoped module-owned registries.

Portia should not expand Core solely to generalize one small teacher-local Actor Directory prematurely.

---

## 9.6 No Core Identity Expansion

Portia v1 does not require Core to add:

* a workspace ID;
* a school ID;
* a teacher ID;
* workspace-wide student identity;
* cross-roster student reconciliation;
* institution-wide person records;
* organization-scoped work roots;
* or cross-teacher authorization.

Adding those concepts would materially expand the scope of Paper Data Suite and is outside the current product model.

---

## 9.7 Core-Impact Decision

The architectural decision for Portia v1 is:

> Use Core’s existing class-qualified work and routing contracts unchanged. Implement the limited workspace-scoped Actor Directory as a documented Portia-owned convention. Do not request broader Core identity or storage changes unless another suite module later demonstrates the same requirement.
